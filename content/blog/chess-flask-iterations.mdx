# Chess Flask

Chess Flask is a good example of a project that benefited from keeping the presentation layer small and the engine experimentation isolated.

The move-generation logic lived in a separate Flask service, while the UI in this repository focused on testing, playing, and comparing versions.

## Why a separate backend helped

The engine changed more often than the interface. That made a clean API boundary useful.

- the chess bot could evolve in its own repository
- the frontend stayed stable
- versioned endpoints made comparisons easier

## Iteration mattered more than polish

The project followed an iterative approach inspired by Sebastian Lague's chess series, but the useful lesson was broader than chess: keep the feedback loop short when the core problem is algorithmic.

### Better search beats premature UI complexity

Most of the meaningful progress came from improving search and evaluation rather than redesigning controls. The frontend only needed enough structure to make those improvements visible and testable.

## A practical boundary

Separating the interface from the engine also made it easier to reason about failure:

```python
@app.get("/move/v3")
def move_v3():
    return {"uci": choose_move(version=3)}
```

If a bot behaved poorly, the debugging question stayed narrow. Was the issue in evaluation, move ordering, search depth, or the contract between frontend and engine?

## What I would improve next

I would add stronger instrumentation around the engine versions so regressions were easier to compare over time. Iteration is most effective when the evidence is easy to inspect.
